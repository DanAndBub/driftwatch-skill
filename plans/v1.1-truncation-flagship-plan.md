# Driftwatch v1.1 — Truncation Flagship Plan

**Created:** 2026-03-23
**Status:** PLANNED
**Goal:** Make Driftwatch the definitive silent truncation detection tool for OpenClaw workspaces
**Repo:** https://github.com/DanAndBub/driftwatch-skill.git

---

## Context

Driftwatch v1.0 ships a working config health scanner with 4 modules: truncation, compaction (anchor health), hygiene, and config check. Community feedback confirms **silent truncation tracking** is the highest-value feature — it solves a pain point nobody else addresses. Operators don't know their bootstrap files are being silently cut until agent behavior degrades.

**Decision:** Cut the config check module (weak, shallow, creates confusion). Double down on truncation as the flagship capability.

## Phase 2: Drift Tracking (Core New Feature)

**Priority:** This is the main value add. Everything else builds on it.

### Concept
Store scan results over time so operators can see how their workspace is changing. Instead of "AGENTS.md is at 32%" it becomes "AGENTS.md grew 1,200 chars this week and is trending toward the limit."

### Architecture

```
~/.driftwatch/
├── history/
│   ├── 2026-03-23T054335Z.json    # Full scan result, timestamped
│   ├── 2026-03-24T120000Z.json
│   └── ...
└── config.json                     # Optional: retention policy, alert thresholds
```

### Tasks
1. **Add `--save` flag to `scan.py`** — after running scan, persist full JSON to `~/.driftwatch/history/{timestamp}.json`
2. **Add `--history` flag** — reads stored scans and produces a diff/trend report
3. **Trend calculation module (`scripts/trends.py`)**:
   - Load last N scans (configurable, default 10)
   - For each bootstrap file: calculate char count delta, growth rate (chars/day)
   - Flag files with sustained growth trends
   - Output: `trends` section in JSON with per-file growth data
4. **History retention** — auto-prune scans older than 90 days (configurable in `~/.driftwatch/config.json`)
5. **First-run handling** — if no history exists, save current scan and note "baseline established, trends available after 2+ scans"

### Output Format (new `trends` section)
```json
{
  "trends": {
    "scans_analyzed": 10,
    "time_span_days": 14,
    "files": [
      {
        "file": "AGENTS.md",
        "current_chars": 6325,
        "oldest_chars": 5100,
        "delta": 1225,
        "growth_rate_chars_per_day": 87.5,
        "days_to_limit": 156,
        "trend": "growing"
      },
      {
        "file": "MEMORY.md",
        "current_chars": 5425,
        "oldest_chars": 5400,
        "delta": 25,
        "growth_rate_chars_per_day": 1.8,
        "days_to_limit": 8097,
        "trend": "stable"
      }
    ],
    "aggregate": {
      "current_total": 27855,
      "growth_rate_chars_per_day": 120,
      "days_to_aggregate_limit": 1017,
      "trend": "stable"
    }
  }
}
```

### Trend Classifications
- **stable**: growth rate < 50 chars/day (effectively flat)
- **growing**: 50-200 chars/day (normal editing, monitor)
- **accelerating**: >200 chars/day or rate increasing over last 3 scans (flag to operator)
- **shrinking**: negative growth (operator cleaned up — good news)

### Acceptance Criteria
- `--save` persists scans to `~/.driftwatch/history/`
- `--history` produces trend report from stored data
- Graceful handling of 0 or 1 stored scans
- History auto-prunes per retention policy
- Tests for trend calculations with fixture data

## Phase 3: Truncation Simulation (Unique Value Prop)

**Priority:** This is what makes Driftwatch irreplaceable. Nobody else shows you exactly what disappears.

### Concept
OpenClaw truncates the middle of files that exceed 20K chars — keeps the first 14K (head zone) and last 4K (tail zone), cuts everything between. Show operators exactly which lines and sections would vanish if their file crosses the limit.

### Tasks
1. **New module: `scripts/simulation.py`**
2. **For each file, calculate the "danger zone"** — the content between char 14,000 and (total_chars - 4,000)
   - If file is under 20K: show what's currently in the danger zone (would be cut first if file grows)
   - If file is over 20K: show what IS being cut right now
3. **Section-level mapping** — parse markdown headings in the danger zone and report which sections are at risk
   - "If AGENTS.md grows to 20K, the following sections would be truncated: ## Delegation Templates (lines 45-89), ## Two-Tier QA (lines 90-105)"
4. **Severity scaling**:
   - File at <60% of limit: no simulation needed (skip for brevity)
   - File at 60-80%: "approaching — these sections would be first cut"
   - File at 80-95%: "at risk — these sections are in the truncation danger zone"
   - File at >100%: "TRUNCATED NOW — these sections are being cut"

### Output Format
```json
{
  "simulation": {
    "files": [
      {
        "file": "AGENTS.md",
        "percent_of_limit": 31.6,
        "simulation_needed": false,
        "note": "Well under limit, no truncation risk"
      },
      {
        "file": "AGENTS.md",
        "percent_of_limit": 85.0,
        "simulation_needed": true,
        "danger_zone": {
          "start_char": 14000,
          "end_char": 13000,
          "start_line": 350,
          "end_line": 480,
          "sections_at_risk": [
            { "heading": "## Delegation Templates", "line": 352, "chars": 2400 },
            { "heading": "## Two-Tier QA", "line": 410, "chars": 450 }
          ]
        },
        "recommendation": "Move critical content above line 350 or below line 480 to protect it from truncation"
      }
    ]
  }
}
```

### Acceptance Criteria
- Only runs simulation for files at >60% of limit
- Correctly identifies head zone (14K), tail zone (4K), and danger zone
- Maps markdown headings in the danger zone
- Provides actionable line numbers
- Tests with fixture files at various percentages

## Phase 5: Cron-Friendly Alert Mode (Automation)

**Priority:** Natural extension of drift tracking. Ship last.

### Concept
A `--check` mode designed for automated monitoring. Pairs with OpenClaw cron jobs.

### Tasks
1. **`--check` flag** — runs scan, checks against thresholds, exits with appropriate code:
   - Exit 0: all healthy
   - Exit 1: warnings present
   - Exit 2: criticals present
2. **`--check --save`** — save + check in one pass (the cron use case)
3. **Threshold config** in `~/.driftwatch/config.json`:
   ```json
   {
     "alert_thresholds": {
       "per_file_warning_percent": 70,
       "per_file_critical_percent": 90,
       "aggregate_warning_percent": 60,
       "aggregate_critical_percent": 80,
       "growth_rate_warning_chars_per_day": 200
     },
     "retention_days": 90
   }
   ```
4. **One-line summary to stdout** when in check mode:
   - `✓ All clear — 8 files healthy, 18.6% aggregate budget used`
   - `⚠ Warning — AGENTS.md at 82% of limit, growing 150 chars/day`
   - `✗ Critical — MEMORY.md truncated, 2 sections lost`

### OpenClaw Cron Integration Example
```json
{
  "schedule": { "kind": "cron", "expr": "0 9 * * *" },
  "payload": {
    "kind": "agentTurn",
    "message": "Run driftwatch --check --save and report any warnings or criticals"
  }
}
```

### Acceptance Criteria
- Exit codes match severity
- Threshold config is optional (sensible defaults)
- One-line summary is parseable by both humans and scripts
- Tests verify exit codes for each severity level

## Non-Goals (For Now)

- **Web dashboard / hosted service** — keep it local-first, self-contained
- **Multi-agent workspace comparison** — single workspace at a time
- **Token counting** — OpenClaw uses char limits, not tokens. We match their model.
- **Config validation** — cut in Phase 1, not coming back
- **Automatic remediation** — Driftwatch reports, operator decides. Read-only forever.
